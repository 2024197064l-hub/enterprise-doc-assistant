这个项目可以描述为：

企业内部文档问答 + 流程助手系统

这是一个面向企业内部知识管理和流程执行的 AI 助手项目。它允许企业上传内部文档，例如制度文件、SOP、FAQ、培训材料、项目说明、Excel 表格等，然后员工可以像和 AI 聊天一样直接提问。系统会从企业内部文档中检索相关内容，生成带依据的回答，并进一步给出对应的流程建议、责任部门、下一步操作或工单建议。

它不是一个单纯的聊天机器人，而是一个 “企业知识检索 + AI 问答 + 流程执行辅助” 的一体化 MVP。


---

一、项目解决的核心痛点

1. 企业内部信息分散，员工找不到答案

很多公司的制度、流程、项目资料散落在：

Word 文档

PDF 文件

Excel 表格

企业网盘

群公告

邮件

Notion / Confluence / 飞书文档


员工遇到问题时，往往不知道应该去哪里找，也不知道关键词该怎么搜。

比如：

“报销标准在哪里？”
“采购电脑需要谁审批？”
“新人入职第一天要做什么？”
“客户投诉升级流程是什么？”
“VPN 不能用了找谁？”

传统做法依赖人工问同事、翻文档、找管理员，效率低，而且容易得到过期或不一致的答案。

这个项目的核心价值是：把分散的企业文档变成一个可对话的知识库。


---

2. 企业知识存在“知道的人知道，不知道的人永远不知道”的问题

很多流程并不是完全写在系统里，而是沉淀在老员工经验中。新员工、跨部门协作人员、临时项目成员很容易因为不知道规则而反复打扰别人。

这个系统通过文档上传和检索问答，把隐性知识逐步结构化，让员工可以直接问：

“如果我要申请测试服务器，应该怎么做？”

系统不只是回答文档内容，还可以给出：

应该走什么流程

需要准备什么材料

找哪个部门

下一步该做什么

是否需要创建工单



---

3. 普通搜索只能返回文档，不能直接解决问题

企业内部搜索通常只能搜到一堆文件，用户还要自己打开、阅读、判断、整合。

这个项目的目标不是“帮你找到文档”，而是直接回答：

> 根据已有内部文档，结论是什么？我下一步该怎么做？



例如用户问：

“我要采购一台新电脑，流程是什么？”

普通搜索可能返回《采购管理制度.pdf》。

这个项目会返回类似：

“根据采购审批流程，员工需要先提交采购申请，说明用途、预算和型号；直属主管审批后，由行政或 IT 部门确认配置；超过预算阈值时还需要财务审批。建议你下一步准备采购理由、预算金额和设备型号，并提交采购申请。”

这就是从“文档搜索”升级到“任务辅助”。


---

4. 流程执行靠人工记忆，容易遗漏步骤

企业流程通常有多个步骤、多个角色和多个条件分支。比如采购、报销、请假、权限申请、IT 支持、客户投诉升级等。

员工可能知道“大概该申请”，但不知道：

先找谁

填什么表

多久能完成

哪些情况需要额外审批

缺少材料会不会被退回


所以项目加入了“流程助手”模块，用来把问答结果进一步转化成可执行建议。

它解决的是：不仅告诉员工答案，还告诉员工下一步动作。


---

二、项目的核心定位

这个项目可以定位为：

> 一个轻量级企业内部 AI 知识库与流程助手，支持上传内部文档，自动构建可检索知识库，并通过 RAG 问答和流程模板帮助员工快速获取答案、理解制度并执行下一步操作。



更产品化一点的说法是：

> Enterprise Doc Assistant 是一个面向中小企业和团队的内部知识问答系统。它通过文档解析、文本切分、向量/关键词检索和大模型生成，将企业静态文档转化为可交互、可追溯、可执行的 AI 助手。




---

三、项目的核心逻辑流

项目的核心逻辑可以分成五个阶段：

阶段 1：企业文档上传

管理员或员工上传内部文档，例如：

公司制度

IT 支持手册

HR 政策

采购流程

报销规范

客服 SOP

项目文档

Excel 数据表


系统接收文件后，会保存文件信息，并进入解析流程。

核心动作：

1. 用户上传文件


2. 后端接收文件


3. 判断文件类型


4. 提取文本内容


5. 存储文档元数据




---

阶段 2：文档解析与切分

文档不能直接整体丢给 AI，因为文件可能很长，超过上下文限制，而且检索效率低。

所以系统会把文档内容拆成多个小片段，也就是 chunks。

例如一份 30 页的制度文档，会被切成若干段，每段保留：

所属文档

原始文件名

片段文本

片段位置

创建时间


这样后续用户提问时，系统不需要读取整篇文档，而是只检索最相关的片段。

核心动作：

1. 提取文档全文


2. 清洗无效空白


3. 按长度切分文本


4. 保存文档片段


5. 建立可检索索引




---

阶段 3：用户提问与检索

员工在前端输入问题，例如：

“VPN 连不上怎么办？”

系统会先把问题拿去和已有文档片段做相似度匹配，找出最可能包含答案的内容。

在这个 MVP 版本里，基础能力是本地 TF-IDF 检索。它不依赖云端 embedding 服务，所以即使没有配置大模型 API Key，也可以运行。

如果后续扩展，可以替换为：

OpenAI Embeddings

bge-m3

text-embedding-3-small

Qdrant

Milvus

pgvector

Elasticsearch hybrid search


核心动作：

1. 接收用户问题


2. 在文档片段中检索相关内容


3. 选出 Top-K 相关片段


4. 将片段作为上下文传给回答模块




---

阶段 4：RAG 回答生成

RAG 的意思是 Retrieval-Augmented Generation，即“检索增强生成”。

系统不是让大模型凭空回答，而是先从企业文档中找依据，再基于这些依据生成回答。

逻辑是：

用户问题
  ↓
检索相关文档片段
  ↓
拼接成上下文
  ↓
交给回答模块
  ↓
生成答案 + 引用来源 + 下一步建议

如果没有配置 LLM API Key，系统可以返回基于检索内容的摘要式回答。

如果配置了 OpenAI-compatible API Key，系统会调用大模型生成更自然、更完整的回答。

这一步的关键是：回答必须基于内部文档，而不是模型自由发挥。


---

阶段 5：流程助手生成下一步动作

企业内部问题很多并不是单纯知识问答，而是流程问题。

例如：

“我要申请新电脑” “客户投诉要升级” “我要开通系统权限” “我想请年假” “服务器挂了怎么办”

这类问题需要的不只是解释，而是操作路径。

所以项目加入了流程助手逻辑。它会根据问题和检索到的文档内容，识别用户可能涉及的流程类型，并给出：

流程名称

适用场景

当前建议步骤

需要准备的信息

相关负责人或部门

是否建议创建工单

工单标题和描述建议


例如：

流程类型：IT 支持
建议动作：
1. 确认问题类型：VPN 无法连接
2. 收集设备信息、网络环境、错误截图
3. 提交 IT 支持工单
4. 如影响多人办公，标记为高优先级

这让系统从“知识问答工具”变成了“工作流入口”。

四、项目的模块结构

可以理解为四层架构。

1. 前端层

前端负责用户交互，主要包括：

文档上传页面

已上传文档列表

聊天问答界面

回答结果展示

来源片段展示

流程建议展示


用户不需要理解后端检索逻辑，只需要上传文档、输入问题。


---

2. API 服务层

后端使用 FastAPI，负责提供接口，例如：

上传文档接口

文档列表接口

问答接口

流程模板接口

健康检查接口


它是前端和核心 AI 逻辑之间的桥梁。


---

3. 文档处理与检索层

这一层负责把文件变成可搜索的知识库。

包括：

PDF 解析

DOCX 解析

Excel / CSV 解析

TXT / Markdown 解析

文本切分

SQLite 存储

TF-IDF 检索


这是项目最核心的底座。


---

4. AI 问答与流程层

这一层负责根据检索结果生成最终输出。

包括：

检索增强问答

来源引用

回答生成

流程类型判断

下一步动作建议

工单草稿生成



---

五、一次完整请求的执行链路

以“员工问采购电脑流程”为例。

用户输入：

我想申请一台新的开发电脑，需要走什么流程？

系统执行过程：

1. 前端发送问题到后端 /ask 接口

2. 后端接收问题

3. 检索模块在文档片段中搜索：
   - 采购制度
   - IT 设备申请流程
   - 预算审批规则
   - 资产登记规范

4. 系统取回最相关的几个文档片段

5. RAG 模块构造上下文：
   - 用户问题
   - 相关文档片段
   - 回答要求
   - 不得编造规则的约束

6. 如果配置了 LLM：
   - 调用大模型生成回答
   如果没有配置 LLM：
   - 返回基于片段的检索式摘要

7. 流程助手判断该问题属于“采购 / IT 设备申请”流程

8. 系统输出：
   - 直接答案
   - 申请步骤
   - 需要准备的信息
   - 可能涉及的审批人
   - 建议创建的工单标题
   - 引用的来源文档

最终用户看到的是一个可执行答案，而不是一堆文件链接。

六、项目核心价值

这个项目的价值可以概括为三句话：

1. 把企业文档变成可问答的知识库

员工不用再翻文档，而是直接问问题。

2. 把制度内容变成可执行步骤

系统不仅解释规则，还给出下一步操作。

3. 把内部经验沉淀成标准化流程

减少重复咨询，提高新人上手速度，降低流程执行错误。
